# V3 需求文档 - 多玩家可见性

## 意图分析
- **用户需求**: 多玩家实时可见，展示 AgentCore Memory 会话隔离
- **请求类型**: 新功能
- **范围**: 多组件 (Frontend + Game Server + WebSocket)
- **复杂度**: 中等

---

## 功能需求

### FR-1: 玩家连接注册
- Game Server 维护已连接玩家注册表 (WebSocket → player_id + position + character)
- 玩家登录并进入游戏场景后注册到连接表
- 玩家断开连接时从注册表移除

### FR-2: 玩家加入/离开广播
- 新玩家加入时，向所有已连接玩家广播 `player_join` 消息（包含 player_id、位置、角色类型）
- 新玩家收到所有当前在线玩家列表
- 玩家离开时，向所有剩余玩家广播 `player_leave` 消息

### FR-3: 玩家位置同步
- 玩家移动时，Game Server 向其他所有玩家广播位置更新
- 使用静态精灵图显示远程玩家（复用现有男/女角色 spritesheet）
- 不需要同步行走动画，只同步位置和朝向

### FR-4: 远程玩家显示
- 前端创建 RemotePlayer 类，使用静态精灵图渲染
- 显示玩家名称标签
- 远程玩家可通过地图自由移动（位置由服务端广播驱动）

### FR-5: 独立游戏逻辑
- 每个玩家的 NPC 对话、战斗、任务系统完全独立
- 其他玩家看不到某玩家的对话/战斗状态
- AgentCore Memory 按 player_id 隔离会话

---

## 非功能需求

### NFR-1: 实时性
- 位置同步延迟 < 200ms（局域网/同区域）
- 玩家加入/离开通知即时广播

### NFR-2: 可扩展性
- 不限制同时在线玩家数量
- 广播机制需支持 N 个客户端

### NFR-3: 兼容性
- 不影响现有单玩家功能（NPC对话、战斗、任务系统）
- 不影响现有预生成、寒暄等机制

---

## 技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 远程玩家显示 | 静态精灵图 | 复用现有 spritesheet，代码简洁 |
| 玩家交互 | 无交互，纯展示 | demo 展示用途，减少复杂度 |
| NPC/战斗可见性 | 独立不可见 | 聚焦会话隔离展示 |
| 玩家数量 | 不限制 | 广播机制天然支持 |
| 安全规则 | 跳过 | demo/原型项目 |

## 新增 WebSocket 消息协议

### 客户端 → 服务端
| 消息类型 | 参数 | 用途 |
|----------|------|------|
| `player_move` (已有，需扩展) | `{player_id, x, y, direction}` | 玩家移动，触发广播 |

### 服务端 → 客户端
| 消息类型 | 参数 | 用途 |
|----------|------|------|
| `player_join` | `{player_id, x, y, character, name}` | 新玩家加入通知 |
| `player_leave` | `{player_id}` | 玩家离开通知 |
| `player_moved` | `{player_id, x, y, direction}` | 其他玩家位置更新 |
| `players_list` | `[{player_id, x, y, character, name}, ...]` | 当前在线玩家列表（新玩家加入时收到） |
